# FND-001 — [accessibility/contrast] Input resting border fails non-text contrast

**Severity:** high
**Project:** R&T Deposit Solutions / R&T Connect
**Location:** AddUser screen → all text inputs and textareas (`.form-control[data-input]`, `.form-control[data-textarea]`). Design source: `Admin/Frame 439.svg`.

## Observed (as designed)
Frame 439 draws every input as a 1px `#D9D9D9` ring on a `#FFFFFF` field, 40px tall, 4px radius.

| Theme | Border | Field | Ratio | SC 1.4.11 (needs 3:1) |
|---|---|---|---|---|
| light | `#d9d9d9` | `#ffffff` | **1.41:1** | fail |
| dark | `#444444` | `#1e1e1e` | **1.71:1** | fail |

The dark value follows because the token this maps to, `--color-border-default`, is overridden to `--rnt-neutral-600` under `[data-theme="dark"]`.

The border is the only thing delimiting the control at rest — the field background is the same white as the page surface behind it, so there is no secondary boundary cue.

## Rule
WCAG 2.2 SC 1.4.11 (Non-text Contrast, Level AA) — visual information required to identify a user interface component and its boundary must have a contrast ratio of at least 3:1 against adjacent colors.

## Recommendation (for design / brand owner)
- **Option A — darken the resting border.** `#767676` on white is 4.54:1; `#6e6e6e` on `#1e1e1e` is 3.27:1. Both clear the 3:1 floor. This is the smallest change that resolves the finding.
- **Option B — give the field a distinct background.** Keep `#D9D9D9` and tint the field (e.g. `--color-background-secondary`), so the control boundary is carried by the fill rather than the stroke.
- **Option C — accept as designed.** Argue that hover (`#444444`) and focus (brand border plus a 2px outline) supply the affordance, and that the resting state is decorative. This is a defensible reading only if the form is never the sole means of identifying an editable region; it does not make the SC pass.

Implementation will not change until design updates the frame or the deviation is signed off.

## Implementation note
Built exactly as designed. `theme/rnt-connect-theme.css` §3.1 resting border set to `var(--color-border-default)` (`#d9d9d9` light / `#444444` dark), with the measured ratios recorded in an inline comment. The matching `.has-accessible-features` override in the focus-neutralizer section was repointed to the same token so HAF and non-HAF screens stay in lockstep — OutSystems UI's HAF rule would otherwise darken the border to `--color-neutral-7` (`#383838`).

Note that the previous value was `--color-neutral-5` (`#757575`, 4.61:1) — it passed SC 1.4.11 but did not match the frame. It was a silent accessibility correction of a design-specified value, which is what this finding replaces.
